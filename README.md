# Imtera — интеграция с Яндекс.Картами

Приложение для подключения карточки организации на Яндекс.Картах и отображения отзывов, рейтинга и счётчиков.

**Стек:** Laravel 12 (API) + Vue 3 (SPA) + Playwright (парсер)

## Возможности

- Авторизация через Laravel Sanctum (SPA, cookie-based)
- Сохранение ссылки на организацию с валидацией
- Парсинг отзывов с Яндекс.Карт (до ~600 штук)
- Кэширование результатов в БД
- Постраничный вывод по 50 отзывов
- Отображение среднего рейтинга, количества отзывов и оценок

## Локальная разработка (без Docker)

### Требования

- PHP 8.3+
- Composer
- Node.js 20+
- SQLite или MySQL

### Установка

```bash
cd backend

cp .env.example .env
php artisan key:generate

# SQLite (по умолчанию)
touch database/database.sqlite

composer install
php artisan migrate --seed

npm install
npm run dev

# В другом терминале
php artisan serve
```

Приложение: http://localhost:8000

### Playwright-парсер (опционально, для полной загрузки отзывов)

```bash
cd backend/parser
npm install
npx playwright install chromium
```

Без Playwright работает HTTP-fallback: загружает первые ~50 отзывов из SSR-разметки страницы.

## Подход к парсингу

### Проблема

У Яндекс.Карт нет публичного API для отзывов. Данные подгружаются динамически при прокрутке, а внутренний endpoint `fetchReviews` защищён CSRF-токенами и подписью запросов.

### Решение: Playwright + кэширование

1. **Playwright (основной метод)** — headless Chromium:
   - Открывает страницу `/reviews/` организации
   - Перехватывает ответы `fetchReviews` через network interception
   - Парсит встроенный JSON из SSR-разметки (первые 50 отзывов)
   - Прокручивает страницу для подгрузки следующих порций
   - Собирает все уникальные отзывы по `reviewId`

2. **HTTP-fallback** — если Playwright недоступен:
   - GET-запрос к странице отзывов
   - Извлечение JSON из `<script>` и microdata из DOM
   - ~50 отзывов (первая порция SSR)

3. **Кэширование на бэкенде**:
   - При сохранении ссылки парсер загружает все доступные отзывы
   - Результат сохраняется в таблицу `reviews`
   - API отдаёт страницы по 50 из кэша — без повторного парсинга
   - Кнопка «Обновить данные» запускает повторный парсинг

### Почему кэш, а не парсинг на каждый запрос

- Парсинг занимает 30–120 секунд (прокрутка, ожидание API)
- Яндекс может заблокировать частые запросы
- Пагинация из БД — мгновенная
- Пользователь видит актуальные данные после явного обновления

## API

| Метод | Endpoint | Описание |
|-------|----------|----------|
| POST | `/api/login` | Вход |
| POST | `/api/logout` | Выход |
| GET | `/api/user` | Текущий пользователь |
| GET | `/api/settings` | Настройки организации |
| PUT | `/api/settings` | Сохранить ссылку + парсинг |
| POST | `/api/settings/sync` | Повторный парсинг |
| GET | `/api/organization/reviews?page=1` | Отзывы (50 на страницу) |

## Что доделали бы при большем времени

- **Очередь задач** — парсинг в фоне через Laravel Queue, чтобы не блокировать UI
- **Периодическое обновление** — cron/scheduler для автоматической синхронизации
- **Прокси-ротация** — для стабильного парсинга при блокировках IP
- **Мониторинг разметки** — алерты при изменении структуры страницы Яндекса
- **Тесты** — unit-тесты парсера на сохранённых HTML-фикстурах, e2e через Playwright
- **Rate limiting** — ограничение частоты парсинга на пользователя
- **Поддержка нескольких организаций** — сейчас одна организация на пользователя
